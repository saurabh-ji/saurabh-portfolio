1. Secure Infrastructure with Bastion Host

Architecture Overview

Implemented a jump server (bastion host) in a public subnet to provide secure SSH access to EC2 instances in private subnets. The solution uses security groups to restrict access only from the bastion host to private instances.

Added Enhancement: Configured AWS Systems Manager Session Manager as a backup access method that doesn't require open SSH ports.

Implementation Steps:

  1. Created a VPC with public and private subnets across two availability zones
  2. Launched an EC2 instance in the public subnet as the bastion host
  3. Configured security groups:
    • Bastion host SG: Allow SSH only from approved IP ranges
    • Private instance SG: Allow SSH only from the bastion host's security group
  4. Set up SSH key pair authentication
  5. Configured NAT gateway for outbound internet access from private instances
  6. Enabled detailed monitoring and CloudWatch logs for the bastion host

Security Benefits:

  • Reduced attack surface by eliminating direct SSH access to private instances
  • Centralized access control through the bastion host
  • All SSH sessions can be logged and audited
  • Private instances have no public IP addresses

2. Remote Access with AWS Client VPN

Architecture Overview

Deployed AWS Client VPN to provide secure remote access to VPC resources without exposing them to the public internet. Integrated with Active Directory for authentication.

Added Enhancement: Implemented split-tunnel routing to optimize bandwidth usage by only routing VPC traffic through the VPN.

Implementation Steps:

  1. Created a certificate authority (CA) using AWS Certificate Manager (ACM)
  2. Configured the Client VPN endpoint in the VPC
  3. Set up authorization rules to control access to specific subnets
  4. Configured security groups to allow necessary traffic from VPN clients
  5. Distributed client configuration files with the appropriate certificates
  6. Enabled connection logging to CloudWatch Logs

Note: Client VPN provides TLS 1.2 encryption by default and supports multi-factor authentication when integrated with AWS Directory Service.

3. Hybrid Cloud with Site-to-Site VPN

Architecture Overview

Established a secure IPSec VPN tunnel between AWS VPC and an on-premises data center using AWS Virtual Private Gateway and customer gateway.

Added Enhancement: Configured BGP routing for dynamic path selection and failover between two tunnels for high availability.

Implementation Steps:

  1. Created a virtual private gateway and attached it to the VPC
  2. Configured the customer gateway with the on-premises VPN device details
  3. Created VPN connections with two tunnels for redundancy
  4. Configured routing (both static and BGP options)
  5. Verified tunnel status and monitored connection metrics in CloudWatch
  6. Tested failover between tunnels by manually bringing one down

Operational Benefits:

  • Encrypted connectivity between cloud and on-premises at 1.25 Gbps per tunnel
  • Automatic renegotiation of VPN tunnels if they go down
  • Integration with existing on-premises security policies
  • Cost-effective alternative to AWS Direct Connect for moderate bandwidth needs

4. Private S3 Access with VPC Endpoint

Architecture Overview

Created a Gateway VPC endpoint for S3 to allow private EC2 instances to access S3 buckets without traversing the public internet, improving security and reducing data transfer costs.

Added Enhancement: Implemented S3 bucket policies and VPC endpoint policies for fine-grained access control.

Implementation Steps:

  1. Identified the route tables for private subnets that needed S3 access
  2. Created a Gateway VPC endpoint for S3 service
  3. Associated the endpoint with the identified route tables
  4. Verified that new routes to S3 were added automatically

5. EC2 Key Pair Recovery Solution

Architecture Overview

Developed a recovery process for EC2 instances when the key pair is lost, using EBS volume attachment to a recovery instance.

Added Enhancement: Automated the recovery process using AWS Systems Manager Automation documents for consistent, auditable recovery operations.

Implementation Steps:

  1. Stopped the affected EC2 instance (can't modify volumes of running instances)
  2. Created a temporary recovery instance with a known key pair
  3. Detached the root volume from the original instance
  4. Attached the volume to the recovery instance as a secondary volume
  5. Mounted the volume on the recovery instance and modified the authorized_keys file
  6. Unmounted and detached the volume, then reattached to original instance
  7. Started the original instance and verified SSH access
  8. Terminated the recovery instance after successful recovery

Security Best Practice: After recovery, rotate the key pair and consider implementing AWS Systems Manager for keyless access to reduce dependency on key pairs.

6. Static Website Hosting on S3

Architecture Overview

Deployed a static website using Amazon S3 website hosting feature with CloudFront CDN for global distribution and Route 53 for DNS.

Added Enhancement: Implemented CI/CD pipeline using AWS CodePipeline to automatically deploy changes from GitHub repository to S3.

Implementation Steps:

  1. Created an S3 bucket named to match the intended website domain
  2. Uploaded static website files (HTML, CSS, JavaScript, images)
  3. Configured the bucket for static website hosting in properties
  4. Set bucket policy to make objects publicly readable
  5. Created CloudFront distribution with the S3 bucket as origin

Performance Benefits:

  • 99.9% availability SLA with S3

  • Cost-effective storage with S3's pay-as-you-go model
  • Built-in DDoS protection with AWS Shield Standard

8. Server Management with Lambda

Architecture Overview

Created a serverless solution using AWS Lambda to stop EC2 instances based on specific criteria, triggered by CloudWatch Events on a schedule.

Added Enhancement: Extended functionality to automatically tag instances before stopping and send notification via SNS.

Implementation Steps:

  1. Created IAM role with permissions to describe and stop EC2 instances
  2. Wrote Python Lambda function using Boto3 to:
    • Identify instances with specific tags (e.g., "AutoStop=true")
    • Verify instance state before attempting to stop
    • Apply "StoppedByLambda" tag with timestamp
    • Publish notification to SNS topic
  3. Created CloudWatch Events rule on cron schedule (e.g., daily at 7 PM)
  4. Configured the rule to trigger the Lambda function
  5. Tested functionality by manually triggering the Lambda

Cost Optimization:

  • Automated stopping of non-production instances during off-hours
  • Potential savings of 65-75% on EC2 costs for dev/test environments
  • Serverless execution means no infrastructure to manage
  • Pay only for the milliseconds when Lambda is executing

9. Monitoring with CloudWatch Alarms

Architecture Overview

Configured comprehensive monitoring for EC2 instances with CloudWatch alarms that trigger SNS notifications when thresholds are breached.

Added Enhancement: Implemented EC2 Auto Recovery actions for system status check failures.

Implementation Steps:

  1. Created SNS topic for notifications with email subscriptions
  2. Configured CloudWatch alarms for:
    • CPU utilization > 80% for 5 minutes
    • Status check failures (system and instance)
    • Memory utilization (using custom metrics from CloudWatch agent)
    • Disk space thresholds
  3. Set up EC2 Auto Recovery action for system status check failures
  4. Configured alarm actions to send SNS notifications
  5. Created dashboard to visualize all alarms and metrics

Pro Tip: For production workloads, consider setting up different notification channels (SMS, Slack, PagerDuty) based on alarm severity levels.

10. Domain Hosting with Route 53

Architecture Overview

Managed DNS for a domain using Route 53 with health checks and failover routing policies for high availability.

Added Enhancement: Implemented DNSSEC for additional security against DNS spoofing attacks.

Implementation Steps:

  1. Registered a new domain or transferred existing domain to Route 53
  2. Created hosted zone for the domain
  3. Configured record sets:
    • A record for web traffic pointing to ALB
    • MX records for email
    • TXT records for domain verification
    • CNAME records for subdomains
  4. Set up health checks for critical endpoints
  5. Configured failover routing between primary and secondary regions
  6. Enabled DNSSEC signing for the hosted zone
  7. Configured domain renewal auto-pay to prevent accidental expiration

DNS Advantages:

  • 100% SLA for Route 53
  • Anycast routing for low-latency DNS resolution
  • Integration with other AWS services through alias records
  • Traffic flow for complex routing scenarios

11. Custom VPC Design

Architecture Overview

Designed and implemented a custom VPC following AWS best practices with public and private subnets across multiple availability zones.

Added Enhancement: Implemented VPC Flow Logs for network traffic monitoring and security analysis.

Implementation Steps:

  1. Created VPC with appropriate CIDR block considering future growth
  2. Designed subnet strategy:
    • Public subnets for internet-facing resources
    • Private subnets for application servers
    • Isolated subnets for databases
  3. Configured route tables with:
    • Internet gateway for public subnets
    • NAT gateway for private subnets
  4. Set up network ACLs for subnet-level firewall rules
  5. Enabled VPC Flow Logs to capture IP traffic data
  6. Configured DHCP options set with custom DNS servers
  7. Peered with other VPCs as needed for inter-VPC communication

Design Consideration: When planning CIDR blocks, ensure there's no overlap with any networks you might need to connect to (on-premises or other VPCs).
